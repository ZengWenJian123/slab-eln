# 实验图片记录管理系统 UI 设计文档

## 1. 文档目标

本文档用于指导 Codex 按照统一的页面结构、交互规则和组件规范，实现一套适用于实验图片记录的软件 Web 前端。

设计原则：

* 严格围绕现有 PRD
* 不新增业务功能
* 优先使用成熟框架和组件
* 页面结构清晰，便于实验人员快速录入和查找
* 表单、表格、详情页、图片页保持统一交互风格

---

# 2. 整体设计原则

## 2.1 用户角色假设

基于你的产品文档，页面主要服务以下角色：

* 管理员
* 实验员
* 只读用户

UI 层只做权限可见性控制，不改变业务流程。

## 2.2 设计风格

建议整体风格为：

* 工具型后台系统
* 信息密度中高
* 注重录入效率和查询效率
* 视觉简洁，减少装饰性元素
* 图片区域突出，但不做复杂视觉设计

## 2.3 布局原则

采用典型后台布局：

* 左侧固定导航
* 顶部全局栏
* 中间内容区
* 页面标题区 + 操作区 + 主内容区
* 列表页以“筛选 + 表格/卡片”结构为主
* 详情页以“基础信息 + 分区卡片”结构为主

---

# 3. 技术框架与组件选型

这部分是给 Codex 的硬约束。

## 3.1 前端框架

建议：

* **Next.js 15+**
* **TypeScript**
* **App Router**

原因：

* 成熟
* 文档完善
* 适合后台管理系统
* 方便后续接口和页面统一组织

## 3.2 UI 组件库

建议主选：

* **Ant Design**

原因：

* 后台系统成熟度高
* 表单、表格、上传、抽屉、弹窗、Tabs、日期选择器都很成熟
* Codex 生成稳定
* 不需要自己拼太多基础组件

不建议当前阶段选 shadcn/ui 作为主组件库，因为它更适合定制型项目，你现在更需要稳定和开发速度。

## 3.3 样式方案

建议：

* **Ant Design 原生样式**
* 必要时少量配合 CSS Modules

理由：

* 避免额外引入 Tailwind 带来风格分裂
* 降低 Codex 拼装复杂度

## 3.4 表单方案

建议：

* **Ant Design Form**
* **Zod** 用于服务端 / 接口校验

前端页面层面优先使用 Antd Form，避免 React Hook Form 和 Antd 混用导致复杂度上升。

## 3.5 表格方案

建议：

* **Ant Design Table**

## 3.6 图片上传与预览

建议：

* **Ant Design Upload**
* **Ant Design Image / Image.PreviewGroup**

## 3.7 日期时间

建议：

* **dayjs**
* Ant Design DatePicker

## 3.8 状态提示

建议：

* Ant Design Tag
* Ant Design Badge
* Ant Design Message
* Ant Design Modal
* Ant Design Popconfirm

# 7. 核心交互规则

## 7.1 列表页统一规则

所有列表页统一采用：

* 页面标题
* 主操作按钮（新建）
* 筛选区
* 表格区
* 分页区

## 7.2 新建/编辑页统一规则

所有新建页统一采用：

* 基础信息卡片
* 业务参数卡片
* 状态与备注卡片
* 页面底部固定操作按钮

按钮顺序统一为：

* 保存
* 取消

## 7.3 状态显示规则

状态字段统一使用颜色标签：

* 成功：绿色 Tag
* 失败：红色 Tag
* 草稿/未完成若后端有此字段再显示，否则不加

## 7.4 失败拦截规则

所有工序模块统一：

* 选择“失败”后，立刻显示“失败原因”
* 未填写失败原因时，保存按钮提交失败并提示

## 7.5 自动计算字段规则

自动计算字段全部：

* 只读展示
* 与输入字段同行或下一行展示
* 计算异常时显示灰色占位或红色提示
