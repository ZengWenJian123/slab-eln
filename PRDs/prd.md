实验室电子记录与信息管理系统 (ELN & LIMS) PRD
1. 项目概述
本项目旨在开发一套适用于小型实验室的 Web 端电子记录与信息管理系统。核心目标是实现合金非晶丝从“成分设计 -> 电弧熔炼 -> 拉丝及玻璃退除 -> 后处理 -> 性能测试”全生命周期的数字化记录、严格的样本溯源（UID）以及多源数据的自动化整合与清洗。

2. 核心系统架构
后端框架建议: Python (FastAPI 或 Django)，便于原生集成现有的 Python 数据处理脚本。

3. 核心功能模块划分
3.1 样本追踪与命名引擎 (UID System)
底层逻辑: 每一个物理实体（不管是锭子、长丝还是2cm短丝）在系统中生成时，立刻赋予一个唯一的 UUID v4 作为主键（无具体业务含义，仅供数据库关联）。系统提供打印条码/二维码功能。

显示别名引擎 (Display Name Generator): 根据用户输入的规则，前端自动拼接出人类可读的语义化名称，仅用于展示和沟通。

推荐规则: [成分代号]-[熔炼批次]-[拉丝批次]-[丝状态+直径]-[后处理代号]-[样本序号]

示例: H8-M0325-L0330-GR32-DC20-1 (H8成分-3月25日熔炼-3月30日拉丝-裸丝32um-20mA直流退火-1号样品)。

优势: 后台关系不依赖这串长字符，即使因为人为原因修改了某个工序参数导致别名变化，底层的 UUID 映射依然稳固，不会发生数据断链。

3.2 实验工序全流程模块
所有工序模块必须具备**“阴性数据”强制拦截**机制：若标记为“失败”，必须强制填写/选择失败原因字典（如：坩埚破裂、断丝、未成相）才能保存。

工序一：成分设计 (Alloy Design)
输入: 元素库多选（默认 Co, Fe, Si, B, Cr, Cu, Nb 等）。

验证规则: 各元素原子分数累加必须严格等于 100%，否则前端阻止提交。

输出: 自动生成全局唯一的成分代号（如 H1, H2）。

工序二：电弧熔炼 (Arc Melting)
关联: 选择已有的成分代号。

输入: 熔炼日期、目标熔炼重量（必填，系统根据目标重量和原子百分比自动反算各元素应称重质量）、熔炼后锭子质量、熔点（选填）。

计算逻辑: 熔炼损耗率 = (目标熔炼重量 - 熔炼后锭子质量) / 目标熔炼重量 * 100%

输出: 熔炼批次号（规则：M + MMDD，如 M0325）。

工序三：拉丝及玻璃退除 (Melt Spinning & Glass Removal)
关联: 选择熔炼批次号（自动带出成分代号）。

输入: 玻璃管直径、放料质量、拉丝温度、卷绕转速 (rpm)、冷却水温度、负压 (kPa)、包覆丝直径 (um)、裸丝直径 (um)、玻璃腐蚀时间。

扩展验证: 若拉丝温度异常低于物料熔点，系统触发高亮警告。

计算逻辑: 玻璃层厚度 = (包覆丝直径 - 裸丝直径) / 2

状态分支设定: * 定义形态为 GC (玻璃包覆) 或 GR (裸丝)。

提供选项：“是否进行磁学等表征（Br, Bs, Hc）？”，若勾选，则跳转至性能记录模板。

输出: 拉丝批次号（规则：L + MMDD，如 L0330）。

工序四：样品制备态及后处理 (Post-treatment)
前置判断: 选择样品批次，确认状态(GC/GR)及直径。询问“是否已焊接于2cm焊板？”

是 (as态): 冻结拉伸相关的退火选项。

否: 开放所有退火选项。

后处理参数矩阵 (动态表单):

温度退火: 温度 (℃)、保温时间 (h)。

电流退火: 类型 (直流/脉冲)、电流密度 (A/mm²)。自动计算: 电流大小(mA) = 电流密度 * π * (裸丝直径/2)²。通电时间 (min)、占空比 (%)、频率 (Hz)。

拉伸退火: 拉伸配重 (g)。自动计算: 拉伸力(MPa) = (拉伸配重 * 9.8) / 截面积。拉伸时间 (min)。

复合退火: 温度+拉伸、电流+拉伸。

磁场退火 (2cm专属): 增加磁场方向 (轴向/横向)、磁场大小 (Oe)。

工序五：性能检测与数据结合 (Data Pipeline)
原始数据映射: 用户在上传测试设备生成的 CSV/TXT 原始数据时，需扫描或输入样品的 UID/别名。系统将原始文件自动存入 MinIO，并将文件路径与该 UUID 绑定。

自动化清洗 (Py Script Hook): 提供执行入口，调用既有的 Python 脚本读取 MinIO 中的原始数据，提取关键测试结果（如 GMI ratio 最大值、特征阻抗等），并将这些提取出的结构化指标写回 MongoDB，与样品的性能看板绑定。

4. 建议补充的辅助功能 (供 AI 理解系统的健壮性)
数据字典管理器 (Data Dictionary Manager):
提供一个可视化的后台面板，专门用于维护“下拉菜单”中的选项（如：常见的失败原因、使用的测试仪器型号、实验人员名单）。确保所有录入项属于有限集合，避免错别字导致后期 ML 模型读取失败。

动态表单引擎 (Dynamic Schema Engine):
对于工序参数的灵活化需求，建议让 AI 采用基于 JSON Schema 的表单渲染器（如 Formily 或 Vue Formulate）。这允许管理员在不修改数据库列结构的情况下，直接在网页上拖拽添加一个新的参数（如“新增一种退火气体氛围”），新参数将以键值对形式存入 MongoDB。

脚本运行监控 (Script Runner Dashboard):
因为需要接入 Python 脚本，建议系统内嵌一个任务队列（如 Celery 或 RQ）。在前端展示处理状态（“排队中”、“处理成功”、“脚本报错”），方便排查原始数据格式不匹配导致的代码崩溃。
